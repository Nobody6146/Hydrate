Learning track

Below is a guided learning track to follow to learn how to use Hydrate

  • Review models and thow they work
  • Review the types of events and when they fire
  • Learn field expression syntax
  • Learn the h-model, h-property, and h-attribute attributes which are the bread and butter attributes you'll use the most
  • Learn about the element callback attributes h-on, h-input, and h-event
  • Learn the basic templating attributes h-template, h-component, h-init
  • Learn how to build logic behind your components
  • Learn client-side routing attribtues and how routing works
  • Fill in any remaining holes

Field Expression Syntax

This syntax refers to the manner in which a framework attribute has it's argument list formatted.

Usage:

  • This syntax can be used to help specify the name of some type of field and provide a JS expression to work with that field
  • Any single command JS will work here as the expression.
  • This syntax also allows you to process multiple field expression pairing to perform multiple actions on a framework attribute that accepts this syntax.
  • A semi colon at the end of the command will deliminate the begging of the next field and expression pairing
  • All valid JS is supported inside this command including variables, variable/object literals, and expressions
  • The only JS syntax not excepted inside the expression body is comments as they can not be parsed
  • This syntax is clear and easy to read and should look fimilar as it's structured like you'd write inline css

Syntax: ATTRIBUTE_NAME="FIELD_NAME_1: EXPRESSION; FIELD_NAME_2: EXPRESSION;..."

  • ATTRIBUTE_NAME - The name of the framework attribute you're using
    • Examples:
      • h-property="value: text"
      • h-attribute="name: 'dog'"
      • h-property="name: 'dog'; value: text"
  • FIELD_NAME - The name of the field that we want to be manipulating
    • Examples:
      • h-property="value: text"
      • h-property="textContent: 'dog'"
  • EXPRESSION - The JS expression to be resolved as the value of the property
    • The terminating semicolon is optional for the last field and expression pair
    • Examples:
      • h-property="value: 1"
      • h-property="value: 1;"
      • h-property="value: 2 + 2"

Features:

  • This expression resolving by the framework injects a few varible utilities during resolution of model events
    • $hydrate - points to the instance of the HydrateApp instance controlling the page
      • Example:
        • h-property="value: $hydrate.state('test.text')"
    • $element - points to the component currently responding to the model change
      • Example:
        • h-input="text: $element.value"
    • $detail - Points to the "detail" property of the custom JS event that the element is responding to
      • Example:
        • h-property="textContent: $detail.prop"
    • $model - Points to the proxy model that is bound to this component
      • Example:
        • h-on="click: $model.text = text + 1"
    • $event - Points to the instance of the JS event generated by the page
      • In the case of using the event listener attribute, $event will be the event (such as oninput) that we received
      • Example:
        • h-on="input: $model.text = $event.data"
    • $state - Points to the state behind the model that is bound to this component
      • Example:
        • <div h-model="default" h-property="innerText: $state.value">
    • $parent - Points to the parent of the model bound to this component
      • Example:
        • <div h-model="default" h-property="innerText: $parent[$modelName]">
    • $modelName - The name of the model bound to this component
      • Example:
        • <div h-model="default" h-property="innerText: $modelName">
    • $id - A function that gets the id attribute value of the passed in element
      • Syntax: function $id(query?:string|HTMLElement)
        • query - A CSS selector or HTMLElement you want the hydrate id value of.
        • If no parameters are passed, the id value will be returned for the component handling the event (this)
      • Example:
        • <div h-model="default" h-id="1" h-property="innerText: $id()"> <div h-model="default" h-id="1" h-property="innerText: $id('div[h-id]')"> <div h-model="default" h-id="1" h-property="innerText: $id($element.parentElement)">
  • By default, expressions being evaluated for this syntax will have the properties of the model state injected as local variables
    • This means that you can access state values directly from the syntax
    • Example:
      • Where our model has the prop "text": h-property="value: text"

Notes:

  • Expressions are ran and evaluated in strict mode

Framework Event Types

Hydrate generates several different types of events that components and event listeners can respond to

  • When referenced via the JS event system, all Hydrate events will have the name formatted "hydrate.[HYDRATE_EVENT_TYPE]"
  • Exampe:
    • hydrate.track
    • hydrate.bind
  • When referenced inside the framework internally, the "hydrate." is dropped for easier referencing
  • The following event definitions will use the JS event type names

hydrate.track

Trigger: When an element goes from being plain HTML to an element that is being tracked by the framework as a component

JS Event Class: HydrateElementTrackingEvent

Notes:

  • At this point, the element has now become a dynamic component and can be optted into being handled by the framework
  • An element with the Model attribute will become tracked whenever it is added to the DOM, or when the model attribute is added
  • The model being bound to the component does not have to be a valid model name in order for this event to fire.
  • Rebinding a new model to the same model name or even changing the model the element is associated with will not retrigger this event. The Model attribute must be completely removed from the element and readded before the even will trigger again.

hydrate.untrack

Trigger: When an element goes from being a component tracked by the framework back to being just a plain HTML element.

JS Event Class: HydrateElementTrackingEvent

Notes:

  • At this point, the element is no longer being controlled by the framework and cannot opt into model changes
  • This would occur when a component has it's Model attribute removed

hydrate.bind

Trigger: When a new model is bound to the framework or a new model is bound to an element.

JS Event Class: HydrateModelEvent

Notes:

  • This event will trigger whether it's the first time a model is bound under that name to the framework, or if a new model is rebound to that name
  • If changes to the Model attribute occur, this would also cause a rebind
  • Reviewing the event target can be useful for determining whether the bind event was generated by the framework or the DOM.
    • Events generated by the framework will always have the targets being the root DOM node being controlled by the framework.
    • If you don't turn the root node into a component, then you can assure any events with the root as it's target originated from the framework.
    • Similarly, any event with a target other than the root node was generated by the element in order to siginal it performing an action.

hydrate.unbind

Trigger: When an existing model is unbound or rebound from the framework. An unbind can also be generated if an existing model property is removed

JS Event Class: HydrateModelEvent

Notes:

  • Rebinding a model with the same name will first generate the Unbind event for the existing model, and then generate a new bind event for the new model data.
  • A model property is considered "removed" if the delete operator is called on the property , or if a set operation is performed on the model that would would replace the state with a value that did not contain the old property

hydrate.set

Trigger: When the set operator is called on a existing model property

JS Event Class: HydrateModelEvent

Notes:

  • A set event means that a bind event would not occur

hydrate.input

Trigger: When a component with the Input Attribute generates an HTML input event

JS Event Class: HydrateModelEvent

hydrate.on

Trigger: When a component with a On Attribute fires the event listener for the event it's listening to

JS Event Class: HydrateElementEventListenerEventDetails

hydrate.routing.start

Trigger: When a request to client side routing starts

JS Event Class: HydrateRouteEvent

hydrate.routing.resolve

Trigger: When you call "resolve" on a client side routing request

JS Event Class: HydrateRouteEvent

hydrate.routing.reject

Trigger: When you call "reject" on a client side routing request

JS Event Class: HydrateRouteEvent

hydrate.mutation.target.added

Trigger: When the target element had a child element added to it

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.target.removed

Trigger: When the target element had a child element removed from it

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.target.attribute

Trigger: When the target element had an attribute change value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.target.characterdata

Trigger: When the target element had it's character data change value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.parent.added

Trigger: When the target element had it's parent element added to the DOM

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.parent.removed

Trigger: When the target element had it's parent element removed from the DOM

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.parent.attribute

Trigger: When the target element had its parent element had an attribute change value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.parent.characterdata

Trigger: When the target element had its parent element change character data value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.child.added

Trigger: When the target element had a child element added to the DOM

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.child.removed

Trigger: When the target element had a child element removed from the DOM

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.child.attribute

Trigger: When the target element ha a child element have an attribute change value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.child.characterdata

Trigger: When the target element had a child element change character data value

JS Event Class: HydrateElementMutationEvent

hydrate.scaffold

Trigger: When an element with the scaffold attribute is being generated by a component

JS Event Class: HydrateScaffoldEvent

Notes: This event is never generated and sent to the DOM. It is an internal event type only handled by the scaffolding element

Framework Element Attributes

Model Binding Attributes

h-model

Usage: Binds the model to the element and turns it into a hydrate component

Syntax: h-model="MODEL_PATH"

  • MODEL_PATH - The model name or "dot operator" path to the property of a model you wish to bind to the component
    • You can access child properties using the "." operator
    • Examples:
      • h-model="person"
      • h-model="person.name"
  • Modeless Components - MODEL_PATH can also be empty. In this scenario, a modeless component will be created.
    • A modeless component will not be able to respond to property changes, as there is no model attached to it
    • Modeless components are useful for performing basic/reusable task on an element that don't depend on receiving state from your application
    • Examples:
      • <div h-model h-component="LogoTemplate">
      • <div h-model h-event="track: console.log('Added element', this)">
  • Model Name Insertion Operator (^)- Normally a "^" is not a vald character for a JS identifier, however, inside the model attribute, it carries special meaning
    • The "^" symbol is the dynamic model name insertion Operator inside of model attribute, and is only allowed to be used inside a model attribute that's part of a template created using the template attribute
    • During component generation, the template is read and the insertion character is replaced with the passed in model name for the component being generated, before the HTML is generated.
    • Say we have the following template and it is being passed in the model name "default": <template h-template="BasicTemplate"> <span h-model="^" h-property="innerText: value"></span> <span h-model="^.value" h-property="innerText: $state"></span> </template>
      • The result HTML inserted into the DOM would be: <div h-model="default" h-component="BasicTemplate"> <span h-model="default" h-property="innerText: value"></span> <span h-model="default.value" h-property="innerText: $state"></span> </template>
    • You can use the insertion name operator any number of times inside a template

Notes:

  • This attribute is REQUIRED for all elements that want to interact with the framework
  • Not every component needs to be hooked up to a global model in order to interact with the framework. For these we can use modeless components
  • You can nest the model name as far as you want into an object. However, because datatype literals cannot be wrapped in a proxy, $model will always be null

h-nested

Usage: Tells the component to listen to nested property changes

Syntax: h-nested

  • Examples:
    • <div>h-model="person.data" h-nested</div>

Notes:

  • This attribute will make the component listen to all properties of the model, no matter how nested
    • Examples: h-model="person.data"
      • person.data.name
      • person.data.name.first
  • This attribute will not make the component listen to any additional event types, just event types of it's nested properties
  • An element with this attribute will only listen to changes to the model or its immediate properties

h-mutation

Usage: Opts the component into responding to the specified mutation events

Syntax: h-mutation="MUTATION_SOURCE_1: LIST_OF_MUTATIONS; MUTATION_SOURCE_2: LIST_OF_MUTATIONS;..."

  • MUTATION_SOURCE - The name of the source of mutation.
    • The possible values are as follows
      • target - The element itself mutated
      • child - The element had a element that mutated
      • parent - The element's parent mutated
    • The attribute uses the Field-Expression Syntax
    • Examples:
      • h-mutation="parent: attribute"
      • h-mutation="child: attribute"
      • h-input="parent: attribute; child: attribute"
    • And empty attribute body is a quick way to specify that the component will respond to all mutation events of elements associated with it
      • h-mutation=""
  • LIST_OF_MUTATIONS - A comma delimited list of mutation types to respond to
    • The possible values are as follows
      • attribute - An element attribute changed
      • added - A child element was added
      • removed - A child element was removed
      • characterdata - The element's character date changed
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • h-mutation="child: attribute"
      • h-property="target: attribute, added, removed"
      • h-property="child: attribute; target: attribute, added, removed"
  • Wild Cards - A wild card (*) can be used in place of some values to mean all sources/mutations
    • A wild card can be used inside the attribute value similar to an empty value to make the component to respond to all mutation events related to elements associated with it
      • h-mutation="*"
    • A wild card can be used in a list of mutations to make the component respond to all mutation types for a source type
      • h-mutation="target: *"

Event types: This attribute will allow components to respond to

  • mutation.target.added
  • mutation.target.removed
  • mutation.target.attribute
  • mutation.target.characterdata
  • mutation.parent.added
  • mutation.parent.removed
  • mutation.parent.attribute
  • mutation.parent.characterdata
  • mutation.child.added
  • mutation.child.removed
  • mutation.child.attribute
  • mutation.child.characterdata

h-mock

Usage: Tells a Hydrate attribute to respond to a model that's different than the model bound to the element.

Syntax: h-mock="HYDRATE_ATTRIBUTE: MODEL_PATH"

  • HYDRATE_ATTRIBUTE - The name of the Hydrate attribute we want to override the model it's listening to
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • <div h-model="app" h-property="textContent: greeting" h-mock="h-property: 'page'"></div>
  • MODEL_PATH - The name of the model we want the attribute to listen to
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • <div h-model="app" h-property="textContent: greeting" h-mock="h-property: 'page'"></div>
      • <div h-model="app" h-property="textContent: greeting" h-mock="h-property: 'page'; h-class: 'page'" h-class="hidden: active"></div>
      • <div h-model="app" h-property="textContent: greeting" h-mock="h-property: 'page'; h-class: $state.activePage" h-class="hidden: active"></div>
      • <div h-model="app" h-property="textContent: greeting" h-mock="*: 'page'"></div>

Notes:

  • This attribute is likely never needed, but there may be some scenarios where you need an element to listen to multiple models (typically a sign of bad design and need to reorganize your models)
  • Another useful scenario is if you need the model the an element points to, to be dynamic such as a runtime expression, come from a model, component, etc.

h-init

Usage: Binds and empty object to the model path if the model doesn't already exist

Syntax: h-init

  • Examples:
      //Binds the a model {} to the model name "app" if the model doesn't exist
    • <div h-model="app" h-init></div>
      //Init can even created nested values //Binds the a model {"page": {}} to the model name "app" if the model doesn't exist
    • <div h-model="app.page" h-init></div>
    • <input h-model="input" h-init h-input="value: $element.value" h-property="value: value"></input>

Notes:

  • This attribute lets you provide an element with state to work with that doesn't have to be bound elsewhere.
  • This attribute is useful in scenarios were the data is originated (bound) from the component itself and not from another JS source.
    • Ex:
    • <app-root h-model="app" h-init></app-root>
  • This attribute is also useful in scenarios where state is needed internally for the element, but other elements or JS code don't need to interact with it.

Event types: This attribute will allow components to respond to

  • track
  • bind
  • routing.start
  • routing.resolve
  • routing.reject
  • mutation.target.attribute

h-bind

Usage: Used to bind a JSON object to a model path from the HTML DOM

Syntax: Remote - <script type="application/json" h-bind="MODEL_PATH: JSON_URL"></script> or Embeded - <script type="application/json" h-bind="MODEL_PATH">JSON_OBJECT</script>

  • MODEL_PATH - The name of the model the payload should be bound to
    • The attribute uses the Field-Expression Syntax
  • JSON_URL - The web url of the JSON file you want to load the payload from
    • The attribute uses the Field-Expression Syntax
  • JSON_OBJECT - The JSON object of the state you want to load
    • All standard JSON formatting rules apply here
  • Examples:
    • Embeded Model Syntax: <script type="application/json" h-bind="app.page">{"data": {"x": 0, "y": 0}}</script>
    • Remote Model Syntax: <script type="application/json" h-bind="app.page: '/customers.json'"></script>

Notes:

  • The bind attribute will not respond to Hydrate events directly (as it is not tracked because it doesn't have a model attribute), but whenever the attributes on the element change or the element is added to the DOM, the binding will trigger
  • The bind attribute only works on a script element with the type set to "application/json".
  • Scripts marked with the bind attribute are not filtered out from component templates, which means you can use them to embed data sources.
  • In the case of server side rendering, the bind attribute is a good way to pass data to Hydrate that was used to create the sever rendered page, so the client-side can then take over.
    • This technique is good for accomplishing partial "hdyration" for client-side rendering
    • To accomplish a full "resumability" approach (server renders content and the client will only rerender on further state changes), the following approaches are available
      • If you want the bind attribute to load the model for later use, but don't want to force a framework update (refreshing DOM, triggering JS callbacks, etc), use the silent attribute
      • If you want the bind attribute to load the model and also framework updates, but prevent certain elements from trigger updates, see the resume attribute to make a specific element resumable

Event types: This attribute will allow components to respond to

  • track
  • mutation.target.attribute

h-silent

Usage: Signals that the bind attribute binds the model data silently (won't trigger framework updates such as DOM updates and callbacks).

Syntax: h-silent

  • Examples:
    • Embeded Model Syntax: <script type="application/json" h-bind="app.page" h-silent>{"data": {"x": 0, "y": 0}}</script>
    • Remote Model Syntax: <script type="application/json" h-bind="app.page: '/customers.json'" h-silent></script>

Notes:

  • This attribute can be used to help create a "resumable" app (an app that doesn't rerender the content from the sever unless further state changes)

Model Binding Conditionals

h-if

Usage: Adds a condition to when the element should respond to an event

Syntax: h-filter="HYDRATE_EVENT_TYPE: CONDITION"

  • HYDRATE_EVENT_TYPE - The name of the Hydrate event we want to restrict when the element responds to it
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • <div h-model="app" h-event="*: console.log('change detected!')" h-if="event: page"></div>
      • <div h-model="app" h-event="*: console.log('change detected!')" h-filter="*: page"></div>
  • PROPERTY_NAME_LIST - The list of the model properties we want to allow the element to respond to
    • The property attribute uses the Field-Expression Syntax to seperate list
    • The property names do not need to be encapsulated in quotes and can be delimited by whitespace and/or ','.
    • Examples:
      • <div h-model="app" h-event="*: console.log('change detected!')" h-if="*: name === 'Home'"></div>
      • <div h-model="app" h-event="*: console.log('change detected!')" h-if="event: name === 'Home'"></div>

Notes:

  • This attribute is useful for restricted the number of events an element has to respond to if only a couple properties are relavent to the element (by default an element responds to all properties of the model)

h-filter

Usage: Filters which model properties the element will respond to

Syntax: h-filter="HYDRATE_EVENT_TYPE: PROPERTY_NAME_LIST"

  • HYDRATE_EVENT_TYPE - The name of the Hydrate event we to only respond to the specified properties
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • <div h-model="app" h-event="*: console.log('change detected!')" h-filter="event: page"></div>
      • <div h-model="app" h-event="*: console.log('change detected!')" h-filter="*: page"></div>
  • PROPERTY_NAME_LIST - The list of the model properties we want to allow the element to respond to
    • The property attribute uses the Field-Expression Syntax to seperate list
    • The property names do not need to be encapsulated in quotes and can be delimited by whitespace and/or ','.
    • Examples:
      • <div h-model="app" h-event="*: console.log('change detected!')" h-filter="*: page"></div>
      • <div h-model="app" h-event="*: console.log('change detected!')" h-filter="*: page, home"></div>
      • <div h-model="app" h-event="*: console.log('change detected!')" h-filter="*: page home; event: about"></div>

Notes:

  • This attribute is useful for restricted the number of events an element has to respond to if only a couple properties are relavent to the element (by default an element responds to all properties of the model)

h-resume

Usage: Makes the element have a resumability behavior by delaying model propagation to the element unless it is interacted with or subsequent model changes occur.

Syntax: h-resume

  • Examples:
    • <div h-model="app.page" h-resume h-property="textContent: greeting">Hello my default greeting</div>

Notes:

  • In short, this model won't respond to track events but will respond to all other events
  • Resumability is useful if you want to pre-render content on the server but not want to cause the UI to re-render unless further state changes occur
  • To fully utilize the "resumability" feature of Hydrate, see the bind attribute on how to bind models from external sources (such as from a server).

Element Manipulation Attributes

h-property

Usage: Used to set a property(s) on a hydrate component

Syntax: h-property="PROPERTY_NAME_1: EXPRESSION; PROPERTY_NAME_2: EXPRESSION;..."

  • PROPERTY_NAME - The name of the property on the HTML element you want to set
    • The property attribute uses the Field-Expression Syntax
    • Examples:
      • h-property="value: text"
      • h-property="textContent: 'dog'"
      • h-property="value: text; textContent: 'dog';"
  • EXPRESSION - The JS expression to be resolved as the value of the property
    • The property attribute uses the Field-Expression Syntax
    • Examples:
      • h-property="value: 1"
      • h-property="value: 2 + 2"

Default Event types: This attribute will respond to the following event types by default

  • track
  • bind
  • set
Routing Event types: This attribute will respond to the following event types if you have the routing attribute
  • routing.start
  • routing.resolve
  • routing.reject
Mutation Event types: This attribute will respond to the following event types if you have the Mutation attribute
  • mutation.target.added
  • mutation.target.removed
  • mutation.target.attribute
  • mutation.target.characterdata
  • mutation.parent.added
  • mutation.parent.removed
  • mutation.parent.attribute
  • mutation.parent.characterdata
  • mutation.child.added
  • mutation.child.removed
  • mutation.child.attribute
  • mutation.child.characterdata

h-attribute

Usage: Used to set a attribute(s) on a hydrate component

Syntax: h-attribute="ATTRIBUTE_NAME_1: EXPRESSION; ATTRIBUTE_NAME_2: EXPRESSION;..."

  • ATTRIBUTE_NAME - The name of the attribute on the HTML element you want to set
    • The attribute uses the Field-Expression Syntax
    • Examples:
      • h-attribute="id: name"
      • h-attribute="class: 'mainTheme'"
      • h-attribute="id: name; class: 'mainTheme'"
  • EXPRESSION - The JS expression to be resolved as the value of the property
    • The attribute uses the Field-Expression Syntax
    • Examples:
      • h-attribute="class: 'mainTheme'"
      • h-attribute="class: `mainTheme-${2}'"

Default Event types: This attribute will respond to the following event types by default

  • track
  • bind
  • set
Routing Event types: This attribute will respond to the following event types if you have the routing attribute
  • routing.start
  • routing.resolve
  • routing.reject
Mutation Event types: This attribute will respond to the following event types if you have the Mutation attribute
  • mutation.target.added
  • mutation.target.removed
  • mutation.target.attribute
  • mutation.target.characterdata
  • mutation.parent.added
  • mutation.parent.removed
  • mutation.parent.attribute
  • mutation.parent.characterdata
  • mutation.child.added
  • mutation.child.removed
  • mutation.child.attribute
  • mutation.child.characterdata

Templating and Reusable HTML/JS Attributes

h-template

Usage: References the inner HTML of the element as a component template that can be later referenced by name

Syntax: <template h-template="TEMPLATE_NAME"> TEMPLATE_HTML </template>

  • TEMPLATE_NAME - The name of the template to reference it
    • Examples:
      • <template h-template="app-root"> Hello World </template>
      • <template h-template="app-root" h-model="app" h-init> Hello World </template>
    • To implicitly generate an instance of a component using the tempalte, you can use the TEMPLATE_NAME as the custom tag name
      • Ex: <app-root></app-root>
      • When this form of component decleration is used, the appropriate component attribute will be added and filled out on the element to track and handle its component generation
  • TEMPLATE_HTML - The HTML you want to be generated
    • Examples:
      • <template h-template="app-root"> Hello World </template>
      • <template h-template="app-root"> <div> Hello World </div> </template>
      • <template h-template="BasicTemplate"> <span h-model="^" h-property="innerText: value"></span> <span h-model="^.value" h-property="innerText: $state"></span> </template>

Notes:

  • Templates are used for defining HTML, CSS, and script bodies for components. See component attribute for further instruction on how to utilize templates
  • In some cases, using the template name as a custom HTML tag is not viable, as the custom tag will always extend the base HTMLElement class. Because of this, the standard HTML element nesting limitations apply and you will not be able to place this custom tag inside certain elements such as select elements
    • To use avoid this issue, you manually put a component attribute on an element to turn it into a component (the custom tag usually does this implicitly for you).
    • There may be other semantic reason for wanting to specifically use a native HTMLElement tag and decorate it instead
  • When inside a template, you can use the model name insertion operator to inject the name of the model name coming from the component. You can read more about it under the model attribute section.
  • Any time "this" is used when evaluating an expression for a Hydrate attribute, "this" will refere to this component object
  • Costume components can also have custom styling and component bodies defined on them. See the following attribtues for more information component attribute or component attribute
  • Attributes included on a template are also copied over to the components being generated (except for the template attribute of course), which is useful for not having to set attributes manually for each instance of a component. Copied attributes however will not override any existing attributes already on the component element.
  • Templates can be lazy loaded with the lazy attribute

h-script

Usage: Denotes a script tag as the class definition for the component

Syntax: <script h-script> CLASS_DEFINITION </script>

  • CLASS_DEFINITION - The class definition of the underlying component
    • This class body's name is not important
    • Examples:
      • class Component() {}
    • The component object created for each instance will contain the following properties:
      • this.$hydrate - { get; }
        • contains a reference to the HydrateApp instance the component is bound to
      • this.$root - { get; }
        • contains a reference to the root element this component is attached to
      • this.$state - { get; }
        • contains a reference to the state value of the model data bound to the component when it was created
      • this.$model - { get; set; }
        • contains a reference to the state value of the model data bound to the component when it was created
      • this.$modelPath - { get; }
        • contains a reference to the model path of the model data bound to the component when it was created
      • this.$dependency - function<<T>(definition:new (...args) => T): T}
        • Will grab the dependency of the requested type for that component
        • Dependencies requested this way will automatically be disposed of when the component is destoryed
      • this.onInit - function(eventDetails:HydrateEventDetails): void
        • This method is called when the component is initialized (built/rendered) for the first time
        • Rerenders of the component don't require it to be reinitialized, and therefore this method is only called once for the element
        • If not implemented in the component body, this call will never be made
      • this.onPreRender - function(eventDetails:HydrateEventDetails): void
        • This method is called when the component is about to rerender (including for the first time)
        • If not implemented in the component body, this call will never be made
      • this.onPostRender - function(eventDetails:HydrateEventDetails): void
        • This method is called when the component has finished rendering (including for the first time)
        • If not implemented in the component body, this call will never be made
      • this.onDestroy - function(): void
        • This method is called when the component is about to be destroyed, but before it becomese untracked and it's references disposed
        • If not implemented in the component body, this call will never be made

Notes:

  • The class definition should be they only code inside the script or you will get an error.
  • The class definition can also be an old school function instead
  • Any time "this" is used when evaluating an expression for a Hydrate attribute, "this" will refere to this component object
  • Any depedenies created via component.$dependency will automatically be cleaned up for you when the component is destoryed (untracked)
  • If there are multiple script tags with a "h-script" tag, then the last script will win as the component body

Warning:

  • Sometimes using this attribute can cause timing issues when loading a component in some Chromium based browsers, resulting in components not properly linking to elements

h-style

Usage: References the style element that will be used to locally scope the style of the component

Syntax: <style h-style> CSS_RULES </script>

  • CSS_RULES - Any CSS rules you want to use to style the document
    • The local styling is added to the DOM via a style element, so it will apply to the whole document
    • To restrict the styling to the component's contents, use the CSS selector ":root" which will automatically be replaced with a selector for the custom element's tag instead of the document root before it is inserted into the DOM
      • Example: :root > div {color: white} will become -> :app-root > div {color: white}

Notes:

  • If multiple style elements have the "h-style" attribute, the last style will be the locally scoped style (will have it's ":root" selector replaced)
  • Remember, this style is applied globally as a style element to your DOM tree and can interfere with other style sheets

h-component

Usage: Used to generate a component(s) from an HTML template passing in model data

Syntax: h-component="TEMPLATE_SOURCE_TYPE: TEMPLATE_NAME_EXPRESSION"

  • TEMPLATE_SOURCE_TYPE - The type of source the HTML template will come from
    • The possible values are as follows
      • template - The DOM will be queried for a template attribute whos name matches the expression
      • url - A fetch request will be performed to download the HTML for the component
    • The attribute uses the Field-Expression Syntax
    • Examples:
      • h-component="template: 'BasicTemplate'"
      • h-component="url: 'pages/index.html'"
  • TEMPLATE_NAME_EXPRESSION - An expression that results into the string used to find the template
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • h-component="template: 'BasicTemplate'"
      • h-component="template: `${color}Template`"

Default Event types: This attribute will respond to the following event types by default

  • track
  • bind
  • set
Mutation Event types: This attribute will respond to the following event types if you have the Mutation attribute
  • mutation.target.added
  • mutation.target.removed
  • mutation.target.attribute
  • mutation.target.characterdata
  • mutation.parent.added
  • mutation.parent.removed
  • mutation.parent.attribute
  • mutation.parent.characterdata
  • mutation.child.added
  • mutation.child.removed
  • mutation.child.attribute
  • mutation.child.characterdata
Routing Event types: This attribute will respond to the following event types if you have the routing attribute
  • routing.start
  • routing.resolve
  • routing.reject

Notes:

  • A component can be linked to a template without a model. However, the component will not generate and render unless it has a model bound to it.
  • If both the component element and the template definition don't specify a model attribute the component is assumbed to be a modeless component and the appropriate model attribute is added.
  • When this attribute responds to a change, the contents of the component are cleared and regenerated
  • When the model bound to the component is an object, the entire object is passed into component genration and the template is inserted just once
  • When the model bound to the component is an array, the HTML template is inserted for each item in the array being passed in one at a time.
  • A component will not be rerendered if new data is bond to it that does not change it's dimension (from a object to an array or from an array of different size)
  • When inside a template, you can use the model name insertion operator to inject the name of the model name coming from the component. You can read more about it under the model attribute and template attribute section.
  • Components can be lazy loaded with the lazy attribute

Routing:

  • A component with a routing attribute will only generate the component on route changes and not for other event types like it normally would
  • If a component receives a route request for a route that doesn't match, the component's inner HTML will be cleared
  • If an element matches a route and has children, the following happens:
    • There is no child element with a matching routing type - no changes to the component is made
    • There is a child element that matches the routing type - the first child with a matching type will be updated. All other children components will have their inner HTML cleared
  • The same rules apply for an element that contains both the route and routing attribute, except it doesn't affect it's children
  • This is useful for things such as showing different page based on your route or showing loading/error modals on routing

h-source

Usage: Used on a hydrate template to point to a remote (external file) location to where the template body is defined, and should be used to define the element.

Syntax: HTML Template - <template h-template="app-root" h-source="FILE_URL"></script> or Directory Template - <template h-template="app-root" h-source="DIRECTORY_URL"></script> or JS Template - <template h-template="app-root" h-source="FILE_URL"></script>

  • FILE_URL - The URL path to the HTML or JS file containing the template body for the component
    • The contents of the file are identical to if the template was embeded in the DOM
    • The file name will include the file extension
  • DIRECTORY_URL - The directory path to the folder containing all the files that make up the component
    • The component directory will look for the following 3 files for constructing the template
      • index.html - A regular template body to define the markup for the component. You will still need to have this embeded inside a template tag
      • index.css - A css style sheet file defining any custom styling for the component. The ":root" replacement syntax also works here. This would overload the style attribute of the template body
      • index.js - A JS file for definning the logic class behind the component. This would overload the script attribute of the template body
    • If any of the predefined files aren't found, the framework will throw a warning but won't error. You are not required to explicitly define any of these files.
  • Examples:
    • HTML Template Syntax: <template h-template="app-root" h-source="/components/about.html"></script>
    • Directory Template Syntax: <template h-template="app-root" h-source="/components/about"></script>
    • JS Template Syntax: <template h-template="app-root" h-source="/components/about.js"></script>

Notes:

  • When using a remote template, the definition is copied and put into the destination element (element with the source attribute).
  • When using a remote templates, any attributes listed on the source template tag will also be copied to the destination template tag

Notes:

  • The id value starts at "0" for each component and is incremented by 1 for each instance of the id attribute used when inserting the template for each iteration of the components passed in data
    • Ex: The first call would be 0, 1, 2, 3, etc...
  • Each time the component is regenerated from stratch via the component attribute and it's event responses, the id count is reset back to the intial value before generating
  • Each dynamic component's inner HTML has their own id values.
    • This means any two components used throughtout the page can have their own id of value 0, 1, 2, 3, etc.

h-route

Usage: Used to tell a component or child components what route it is listening to

Syntax: h-route="ROUTE_PATH"

  • ROUTE_PATH - A regular-expression type string that represent the route's path
    • Examples:
      • h-route="#about"
      • h-route="#albums/:id"
      • h-route="/index.html

Notes:

  • The path can include Regex
  • All paths are assumed to be absoulte paths (start and end of string)
  • route variables can be used as part of the path
  • The route attribute will apply to the element it's on and the elements children (unless the child has it's own route)
  • For more information about build a route path, see the routing paths section

h-routing

Usage: Used to tell a component which routing events to respond to

Syntax: h-routing="ROUTING_TYPE_1, ROUTING_TYPE_2, ..."

  • ROUTING_TYPE_1 - A delimited string representing all the routing types to respond to
    • The possible values are as follows
      • start - routing start event
      • resolve - routing start event
      • reject - routing start event
    • Any number of routing types can be specified
    • Examples:
      • h-routing="start"
      • h-routing="resolve, reject"
      • h-routing="start resolve"
      • h-routing="start; reject;"

Notes:

  • In order to respond to a routing event, either the target component or it's parent element must have a route attribute attached to it Example:
    • <div h-route="#id" h-routing="start"</div>
    • <div h-route="#id"> <div>h-routing="start"</div> <div>h-routing="endt"</div> </div>
  • Not all attributes work with the routing attribute, so check under the respective attribute to see if it is supported
  • For attributes that support routiing events, they will all respond in some way to a route request even if the route doesn't match. Check the respective attribute details for more information on how it handles routing request
  • Components with the routing attribute will always be initialized (call "onInit") on the routing events they respond to instead of other events (such as bind, set, etc.) like regular components.
    • To easily pass routing data to the component, you can use HydrateRouteRequest.state to get/set the current browser tab history state
      • you can access this route reqeust from the component's "onInit" method as the eventDetails passed will be of type HydrateRouteEventDetails
    • This is intentional so that if you need to pass routing state to the component (such as from your JS client-side routing event listener)
    • Additionally, the component will also be automatically disposed (call "onDispose") and reinitialized (calling "onInit" again), even though the component was already intialized once. This is intentional so that way the component's "onInit" method will be a reliable place for initialinzing the component on every route request.
