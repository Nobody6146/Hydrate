Field Expression Syntax

This syntax refers to the manner in which a framework attribute has it's argument list formatted.

Usage:

  • This syntax can be used to help specify the name of some type of field and provide a JS expression to work with that field
  • Any single command JS will work here as the expression.
  • This syntax also allows you to process multiple field expression pairing to perform multiple actions on a framework attribute that accepts this syntax.
  • A semi colon at the end of the command will deliminate the begging of the next field and expression pairing
  • All valid JS is supported inside this command including variables, variable/object literals, and expressions
  • The only JS syntax not excepted inside the expression body is comments as they can not be parsed
  • This syntax is clear and easy to read and should look fimilar as it's structured like you'd write inline css

Syntax: ATTRIBUTE_NAME="FIELD_NAME_1: EXPRESSION; FIELD_NAME_2: EXPRESSION;..."

  • ATTRIBUTE_NAME - The name of the framework attribute you're using
    • Examples:
      • h-property="value: text"
      • h-attribute="name: 'dog'"
      • h-property="name: 'dog'; value: text"
  • FIELD_NAME - The name of the field that we want to be manipulating
    • Examples:
      • h-property="value: text"
      • h-property="textContent: 'dog'"
  • EXPRESSION - The JS expression to be resolved as the value of the property
    • The terminating semicolon is optional for the last field and expression pair
    • Examples:
      • h-property="value: 1"
      • h-property="value: 1;"
      • h-property="value: 2 + 2"

Features:

  • This expression resolving by the framework injects a few varible utilities during resolution of model events
    • $hydrate - points to the instance of the HydrateApp instance controlling the page
      • Example:
        • h-property="value: $hydrate.state('test.text')"
    • $element - points to the component currently responding to the model change
      • Example:
        • h-input="text: $element.value"
    • $detail - Points to the "detail" property of the custom JS event that the element is responding to
      • Example:
        • h-property="textContent: $detail.prop"
    • $model - Points to the proxy model that is bound to this component
      • Example:
        • h-on="click: $model.text = text + 1"
    • $event - Points to the instance of the JS event generated by the page
      • In the case of using the event listener attribute, $event will be the event (such as oninput) that we received
      • Example:
        • h-on="input: $model.text = $event.data"
    • $state - Points to the state behind the model that is bound to this component
      • Example:
        • <div h-model="default" h-property="innerText: $state.value">
    • $parent - Points to the parent of the model bound to this component
      • Example:
        • <div h-model="default" h-property="innerText: $parent[$modelName]">
    • $modelName - The name of the model bound to this component
      • Example:
        • <div h-model="default" h-property="innerText: $modelName">
    • $script - A function that will return the function denoted with the script attribute who's value matches the inputed string
      • Syntax: function $script(NAME:string)
        • NAME - The name of the corresponds the value of the Script Attribute you want to access
      • Example:
        • <div h-model="default" h-script="calculateSquare" h-property="value: $script('calculateSquare')(value)">
    • $id - A function that gets the id attribute value of the passed in element
      • Syntax: function $id(query?:string|HTMLElement)
        • query - A CSS selector or HTMLElement you want the hydrate id value of.
        • If no parameters are passed, the id value will be returned for the component handling the event (this)
      • Example:
        • <div h-model="default" h-id="1" h-property="innerText: $id()"> <div h-model="default" h-id="1" h-property="innerText: $id('div[h-id]')"> <div h-model="default" h-id="1" h-property="innerText: $id($element.parentElement)">
  • By default, expressions being evaluated for this syntax will have the properties of the model state injected as local variables
    • This means that you can access state values directly from the syntax
    • Example:
      • Where our model has the prop "text": h-property="value: text"

Notes:

  • Expressions are ran and evaluated in strict mode

Framework Event Types

Hydrate generates several different types of events that components and event listeners can respond to

  • When referenced via the JS event system, all Hydrate events will have the name formatted "hydrate.[HYDRATE_EVENT_TYPE]"
  • Exampe:
    • hydrate.track
    • hydrate.bind
  • When referenced inside the framework internally, the "hydrate." is dropped for easier referencing
  • The following event definitions will use the JS event type names

hydrate.track

Trigger: When an element goes from being plain HTML to an element that is being tracked by the framework as a component

JS Event Class: HydrateElementTrackingEvent

Notes:

  • At this point, the element has now become a dynamic component and can be optted into being handled by the framework
  • An element with the Model attribute will become tracked whenever it is added to the DOM, or when the model attribute is added
  • The model being bound to the component does not have to be a valid model name in order for this event to fire.
  • Rebinding a new model to the same model name or even changing the model the element is associated with will not retrigger this event. The Model attribute must be completely removed from the element and readded before the even will trigger again.

hydrate.untrack

Trigger: When an element goes from being a component tracked by the framework back to being just a plain HTML element.

JS Event Class: HydrateElementTrackingEvent

Notes:

  • At this point, the element is no longer being controlled by the framework and cannot opt into model changes
  • This would occur when a component has it's Model attribute removed

hydrate.bind

Trigger: When a new model is bound to the framework or a new model is bound to an element.

JS Event Class: HydrateModelEvent

Notes:

  • This event will trigger whether it's the first time a model is bound under that name to the framework, or if a new model is rebound to that name
  • If changes to the Model attribute occur, this would also cause a rebind
  • Reviewing the event target can be useful for determining whether the bind event was generated by the framework or the DOM.
    • Events generated by the framework will always have the targets being the root DOM node being controlled by the framework.
    • If you don't turn the root node into a component, then you can assure any events with the root as it's target originated from the framework.
    • Similarly, any event with a target other than the root node was generated by the element in order to siginal it performing an action.

hydrate.unbind

Trigger: When an existing model is unbound or rebound from the framework. An unbind can also be generated if an existing model property is removed

JS Event Class: HydrateModelEvent

Notes:

  • Rebinding a model with the same name will first generate the Unbind event for the existing model, and then generate a new bind event for the new model data.
  • A model property is considered "removed" if the delete operator is called on the property , or if a set operation is performed on the model that would would replace the state with a value that did not contain the old property

hydrate.set

Trigger: When the set operator is called on a existing model property

JS Event Class: HydrateModelEvent

Notes:

  • A set event means that a bind event would not occur

hydrate.input

Trigger: When a component with the Input Attribute generates an HTML input event

JS Event Class: HydrateModelEvent

hydrate.on

Trigger: When a component with a On Attribute fires the event listener for the event it's listening to

JS Event Class: HydrateElementEventListenerEventDetails

hydrate.routing.start

Trigger: When a request to client side routing starts

JS Event Class: HydrateRouteEvent

hydrate.routing.resolve

Trigger: When you call "resolve" on a client side routing request

JS Event Class: HydrateRouteEvent

hydrate.routing.reject

Trigger: When you call "reject" on a client side routing request

JS Event Class: HydrateRouteEvent

hydrate.mutation.target.added

Trigger: When the target element had a child element added to it

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.target.removed

Trigger: When the target element had a child element removed from it

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.target.attribute

Trigger: When the target element had an attribute change value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.target.characterdata

Trigger: When the target element had it's character data change value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.parent.added

Trigger: When the target element had it's parent element added to the DOM

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.parent.removed

Trigger: When the target element had it's parent element removed from the DOM

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.parent.attribute

Trigger: When the target element had its parent element had an attribute change value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.parent.characterdata

Trigger: When the target element had its parent element change character data value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.child.added

Trigger: When the target element had a child element added to the DOM

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.child.removed

Trigger: When the target element had a child element removed from the DOM

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.child.attribute

Trigger: When the target element ha a child element have an attribute change value

JS Event Class: HydrateElementMutationEvent

hydrate.mutation.child.characterdata

Trigger: When the target element had a child element change character data value

JS Event Class: HydrateElementMutationEvent

Framework Element Attributes

Model Binding Attributes

h-model

Usage: Binds the model to the element and turns it into a hydrate component

Syntax: h-model="MODEL_PATH"

  • MODEL_PATH - The model name or "dot operator" path to the property of a model you wish to bind to the component
    • You can access child properties using the "." operator
    • Examples:
      • h-model="person"
      • h-model="person.name"
  • Modeless Components - MODEL_PATH can also be empty. In this scenario, a modeless component will be created.
    • A modeless component will not be able to respond to property changes, as there is no model attached to it
    • Modeless components are useful for performing basic/reusable task on an element that don't depend on receiving state from your application
    • Examples:
      • <div h-model h-component="LogoTemplate">
      • <div h-model h-event="track: console.log('Added element', this)">
  • Model Name Insertion Operator (^)- Normally a "^" is not a vald character for a JS identifier, however, inside the model attribute, it carries special meaning
    • The "^" symbol is the dynamic model name insertion Operator inside of model attribute, and is only allowed to be used inside a model attribute that's part of a template created using the template attribute
    • During component generation, the template is read and the insertion character is replaced with the passed in model name for the component being generated, before the HTML is generated.
    • Say we have the following template and it is being passed in the model name "default": <template h-template="BasicTemplate"> <span h-model="^" h-property="innerText: value"></span> <span h-model="^.value" h-property="innerText: $state"></span> </template>
      • The result HTML inserted into the DOM would be: <div h-model="default" h-component="BasicTemplate"> <span h-model="default" h-property="innerText: value"></span> <span h-model="default.value" h-property="innerText: $state"></span> </template>
    • You can use the insertion name operator any number of times inside a template

Notes:

  • This attribute is REQUIRED for all elements that want to interact with the framework
  • Not every component needs to be hooked up to a global model in order to interact with the framework. For these we can use modeless components
  • You can nest the model name as far as you want into an object. However, because datatype literals cannot be wrapped in a proxy, $model will always be null

h-nested

Usage: Tells the component to listen to nested property changes

Syntax: h-nested

  • Examples:
    • <div>h-model="person.data" h-nested</div>

Notes:

  • This attribute will make the component listen to all properties of the model, no matter how nested
    • Examples: h-model="person.data"
      • person.data.name
      • person.data.name.first
  • This attribute will not make the component listen to any additional event types, just event types of it's nested properties
  • An element with this attribute will only listen to changes to the model or its immediate properties

h-mutation

Usage: Opts the component into responding to the specified mutation events

Syntax: h-mutation="MUTATION_SOURCE_1: LIST_OF_MUTATIONS; MUTATION_SOURCE_2: LIST_OF_MUTATIONS;..."

  • MUTATION_SOURCE - The name of the source of mutation.
    • The possible values are as follows
      • target - The element itself mutated
      • child - The element had a element that mutated
      • parent - The element's parent mutated
    • The attribute uses the Field-Expression Syntax
    • Examples:
      • h-mutation="parent: attribute"
      • h-mutation="child: attribute"
      • h-input="parent: attribute; child: attribute"
    • And empty attribute body is a quick way to specify that the component will respond to all mutation events of elements associated with it
      • h-mutation=""
  • LIST_OF_MUTATIONS - A comma delimited list of mutation types to respond to
    • The possible values are as follows
      • attribute - An element attribute changed
      • added - A child element was added
      • removed - A child element was removed
      • characterdata - The element's character date changed
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • h-mutation="child: attribute"
      • h-property="target: attribute, added, removed"
      • h-property="child: attribute; target: attribute, added, removed"
  • Wild Cards - A wild card (*) can be used in place of some values to mean all sources/mutations
    • A wild card can be used inside the attribute value similar to an empty value to make the component to respond to all mutation events related to elements associated with it
      • h-mutation="*"
    • A wild card can be used in a list of mutations to make the component respond to all mutation types for a source type
      • h-mutation="target: *"

Event types: This attribute will allow components to respond to

  • mutation.target.added
  • mutation.target.removed
  • mutation.target.attribute
  • mutation.target.characterdata
  • mutation.parent.added
  • mutation.parent.removed
  • mutation.parent.attribute
  • mutation.parent.characterdata
  • mutation.child.added
  • mutation.child.removed
  • mutation.child.attribute
  • mutation.child.characterdata

Element Callback Attributes

h-input

Usage: A shortcut way to set a model property value based on user input

Syntax: h-input="MODEL_PROPERTY: EXPRESSION; MODEL_PROPERTY: EXPRESSION;..."

  • MODEL_PROPERTY - The name of the property on the model bound to the component
    • The attribute uses the Field-Expression Syntax
    • Examples:
      • h-input="text: $element.value"
      • h-input="character: $event.data"
      • h-input="text: $element.value; character: $event.data"
  • EXPRESSION - The JS expression to be resolved as the value of the property
    • The property attribute uses the Field-Expression Syntax
    • Examples:
      • h-input="text: $element.value"
      • h-input="name: name + $event.data"

Event types: This attribute will respond to the following event types

  • input

h-on

Usage: Adds a custom event listener to the element that will execute the callback

Syntax: h-on="EVENT_TYPE_1: EXPRESSION; EVENT_TYPE_2: EXPRESSION;..."

  • EVENT_TYPE - The name of the event type you want to add the event listener to
    • The attribute uses the Field-Expression Syntax
    • Examples:
      • h-on="click: alert($state)"
      • h-on="input: console.log($event)"
  • EXPRESSION - The JS expression to be resolved as the value of the property
    • The property attribute uses the Field-Expression Syntax
    • Examples:
      • h-on="click: alert($state)"
      • h-on="click: $model.value = 3;"

Notes:

  • The actual event sent via the JS event system will be passed into the function-expression syntax via the $event object

Event types: This attribute will respond to the following event types

  • on

h-event

Usage: Executes the given callback expression when the component responds to the targeted event

Syntax: h-event="EVENT_TYPE: CALLBACK"

  • EVENT_TYPE - The name of Hydrate event type (without the "hydrate.:" prefix)
    • The property attribute uses the Field-Expression Syntax to seperate list
    • For the event type, you can also use a wildcard (*) operator to mean any event that the component responded to
    • Examples:
      • h-event="track: console.log($detail)"
      • h-event="bind: console.log($detail)"
      • h-event="*: console.log($detail)"
  • CALLBACK - Any JS code you want to run as part as the component callback
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • h-event="track: console.log($detail)"
      • h-event="track: $model.value = 0;"

Default Event types: This attribute will respond to the following event types if you have the Mutation attribute

  • track
  • untrack
  • bind
  • unbind
  • set
  • input
  • on
Mutation Event types: This attribute will respond to the following event types if you have the Mutation attribute
  • mutation.target.added
  • mutation.target.removed
  • mutation.target.attribute
  • mutation.target.characterdata
  • mutation.parent.added
  • mutation.parent.removed
  • mutation.parent.attribute
  • mutation.parent.characterdata
  • mutation.child.added
  • mutation.child.removed
  • mutation.child.attribute
  • mutation.child.characterdata
Routing Event types: This attribute will respond to the following event types if you have the routing attribute
  • routing.start
  • routing.resolve
  • routing.reject

Routing:

  • A component with a routing attribute will only trigger on routing event types when the route attribute and routing attribute match the correct criteria
  • If a component receives a route request for a route that doesn't match, the component's inner HTML will be cleared
  • This is useful for doing things such as triggering scripts when routing occurs
  • Visit the routing attribute and other routing sections for additional information

Callback Timing Attributes

h-delay

Usage: For the event types chosen, it delays the execution of the components framework handlers for the specified amount of time

Syntax: h-delay="EVENT_TYPE: EXPRESSION"

  • EVENT_TYPE - The name of Hydrate event type (without the "hydrate.:" prefix)
    • The property attribute uses the Field-Expression Syntax to seperate list
    • For the event type, you can also use a wildcard (*) operator to mean any event that the component responded to
    • Examples:
      • h-delay="track: 500"
      • h-delay="bind: 500"
  • Expression - Any JS code expression resulting in the integer value of how many milliseconds to wait
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • h-delay="track: 500"
      • h-delay="track: $model.delayTime"

Notes: Only one timing change function will be applied to each event type response. They resolve in the following order

  • throttle
  • debounc
  • delay

    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:

Notes: Only one timing change function will be applied to each event type response. They resolve in the following order

h-delay

Usage: For the event types chosen, it debounces the execution of the components framework handlers for the specified amount of time

Syntax: h-debounc="EVENT_TYPE: EXPRESSION"

  • EVENT_TYPE - The name of Hydrate event type (without the "hydrate.:" prefix)
    • The property attribute uses the Field-Expression Syntax to seperate list
    • For the event type, you can also use a wildcard (*) operator to mean any event that the component responded to
    • Examples:
      • h-debounc="input: 500"
      • h-debounc="set: 500"
  • Expression - Any JS code expression resulting in the integer value of how many milliseconds to wait
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • h-debounc="input: 500"
      • h-debounc="input: $model.delayTime"

Notes: Only one timing change function will be applied to each event type response. They resolve in the following order

  • throttle
  • debounc
  • delay

Templating and Reusable HTML/JS Attributes

h-script

Usage: References the inner text of the element as a function decleration which can be referenced with the provided name and called later

Syntax: <script h-script="SCRIPT_NAME"> FUNCTION_DECLERATION </script>

  • SCRIPT_NAME - The name to reference the script by
    • Examples:
      • h-script="Script1"
      • h-script="ExampleScript"
  • FUNCTION_DECLERATION - The decleration of the function
    • Examples:
      • <script h-script="SCRIPT_NAME"> function(firstname, lastname) { console.log(`hello ${firstname} ${lastname}`); } </script>
      • <script h-script="SCRIPT_NAME"> (firstname, lastname) => { console.log(`hello ${firstname} ${lastname}`); } </script>

Notes: Only one timing change function will be applied to each event type response. They resolve in the following order

  • The script can be accessed via the $script function of the field-expression syntax of a framework attribute. E.g.
    • <div h-script="TestScript" h-event="track: $script('TestScript')">
  • The element with the script attribute does not have to be an HTML script element, but it is probably prefered
  • The script decleration should be the only text inside this element's inner text
  • There should only ever be one function per script element
  • The inner contents of the script can be any valid explicit or anonymous function decleration

h-template

Usage: References the inner HTML of the element as a component template that can be later referenced by name

Syntax: <template h-template="TEMPLATE_NAME"> TEMPLATE_HTML </template>

  • TEMPLATE_NAME - The name of the template to reference it
    • Examples:
      • <template h-template="BasicTemplate"> Hello World </template>
      • <div h-template="BasicTemplate"> Hello World </div>
  • TEMPLATE_HTML - The HTML you want to be generated
    • Examples:
      • <template h-template="BasicTemplate"> Hello World </template>
      • <template h-template="BasicTemplate"> <div> Hello World </div> </template>
      • <template h-template="BasicTemplate"> <span h-model="^" h-property="innerText: value"></span> <span h-model="^.value" h-property="innerText: $state"></span> </template>

Notes:

  • Templates are used for defining HTML bodies for components. See component attribute for further instruction on how to utilize templates
  • Although you can use any container HMTL element to hold your template body, the dedicated HTML template tag is the prefered method
  • When inside a template, you can use the model name insertion operator to inject the name of the model name coming from the component. You can read more about it under the model attribute section.

h-component

Usage: Used to generate a component(s) from an HTML template passing in model data

Syntax: h-component="TEMPLATE_SOURCE_TYPE: TEMPLATE_NAME_EXPRESSION"

  • TEMPLATE_SOURCE_TYPE - The type of source the HTML template will come from
    • The possible values are as follows
      • template - The DOM will be queried for a template attribute whos name matches the expression
      • url - A fetch request will be performed to download the HTML for the component
    • The attribute uses the Field-Expression Syntax
    • Examples:
      • h-component="template: 'BasicTemplate'"
      • h-component="url: 'pages/index.html'"
  • TEMPLATE_NAME_EXPRESSION - An expression that results into the string used to find the template
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • h-component="template: 'BasicTemplate'"
      • h-component="template: `${color}Template`"

Default Event types: This attribute will respond to the following event types by default

  • track
  • bind
  • set
Mutation Event types: This attribute will respond to the following event types if you have the Mutation attribute
  • mutation.target.added
  • mutation.target.removed
  • mutation.target.attribute
  • mutation.target.characterdata
  • mutation.parent.added
  • mutation.parent.removed
  • mutation.parent.attribute
  • mutation.parent.characterdata
  • mutation.child.added
  • mutation.child.removed
  • mutation.child.attribute
  • mutation.child.characterdata
Routing Event types: This attribute will respond to the following event types if you have the routing attribute
  • routing.start
  • routing.resolve
  • routing.reject

Notes:

  • When this attribute responds to a change, the contents of the component are cleared and regenerated
  • When the model bound to the component is an object, the entire object is passed into component genration and the template is inserted just once
  • When the model bound to the component is an array, the HTML template is inserted for each item in the array being passed in one at a time.
  • Although you can use any container HMTL element to hold your template body, the dedicated HTML template tag is the prefered method
  • When inside a template, you can use the model name insertion operator to inject the name of the model name coming from the component. You can read more about it under the model attribute and template attribute section.

Routing:

  • A component with a routing attribute will only generate the component on route changes and not for other event types like it normally would
  • If a component receives a route request for a route that doesn't match, the component's inner HTML will be cleared
  • If an element matches a route and has children, the following happens:
    • There is no child element with a matching routing type - no changes to the component is made
    • There is a child element that matches the routing type - the first child with a matching type will be updated. All other children components will have their inner HTML cleared
  • The same rules apply for an element that contains both the route and routing attribute, except it doesn't affect it's children
  • This is useful for things such as showing different page based on your route or showing loading/error modals on routing
  • Visit the routing attribute and other routing sections for additional information

h-duplicate

Usage: Used to duplicate a dynamic component generated from the component attribute any number of times for the model being passed into it

Syntax: h-duplicate="EXPRESSION"

  • EXPRESSION - An expression that results into the number of times you want the template to be inserted
    • The property attribute uses the Field-Expression Syntax to seperate list
    • Examples:
      • h-duplicate="3"
      • h-duplicate="value * 3"

Notes:

  • If the expression cannot be parsed, the number of times is zero
  • When this attribute is used, the model will always be treated like an object (arrays will not get a record for each item). This is useful to force arrays to be treated like an objet

h-id

Usage: Used inside a template marked with the template attribute to generate a unique id for the component

Syntax: h-id

  • This property will be dynamically set with the value assigned when the template is filled out and inserted into the DOM
  • Examples:
    • <template="ArrayTemplate"> <div h-model h-id h-property="innerText: $id()"></div> <template="ArrayTemplate">

Notes:

  • The id value of an element can be referenced via the $id function. Read the field-expression syntax section for further information
  • The id value starts at "1" for each component and is incremented by 1 for each instance of the id attribute used when inserting the template for each iteration of the components passed in data
    • Ex: The first call would be 1, 2, 3, etc...
  • Each time the component is regenerated from stratch via the component attribute and it's event responses, the id count is reset back to 1 before generating
  • Each dynamic component's inner HTML has their own id values.
    • This means any two components used throughtout the page can have their own id of value 1, 2, 3, etc.

Routing Attributes

h-route

Usage: Used to tell a component or child components what route it is listening to

Syntax: h-route="ROUTE_PATH"

  • ROUTE_PATH - A regular-expression type string that represent the route's path
    • Examples:
      • h-route="#about"
      • h-route="#albums/:id"
      • h-route="/index.html

Notes:

  • The path can include Regex
  • All paths are assumed to be absoulte paths (start and end of string)
  • route variables can be used as part of the path
  • The route attribute will apply to the element it's on and the elements children (unless the child has it's own route)
  • For more information about build a route path, see the routing paths section

h-routing

Usage: Used to tell a component which routing events to respond to

Syntax: h-routing="ROUTING_TYPE_1, ROUTING_TYPE_2, ..."

  • ROUTING_TYPE_1 - A delimited string representing all the routing types to respond to
    • The possible values are as follows
      • start - routing start event
      • resolve - routing start event
      • reject - routing start event
    • Any number of routing types can be specified
    • Examples:
      • h-routing="start"
      • h-routing="resolve, reject"
      • h-routing="start resolve"
      • h-routing="start; reject;"

Notes:

  • In order to respond to a routing event, either the target component or it's parent element must have a route attribute attached to it Example:
    • <div h-route="#id" h-routing="start"</div>
    • <div h-route="#id"> <div>h-routing="start"</div> <div>h-routing="endt"</div> </div>
  • Not all attributes work with the routing attribute, so check under the respective attribute to see if it is supported
  • For attributes that support routiing events, they will all respond in some way to a route request even if the route doesn't match. Check the respective attribute details for more information on how it handles routing request
